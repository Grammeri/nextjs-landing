# ADR 0008 — Marketing Site Page Architecture Boundaries

## Status

Accepted

## Context

The Software Forge marketing platform contains multiple types of pages with
different rendering requirements.

During the development of the landing platform several architectural options
were considered, including a universal page engine capable of rendering all
types of pages from structured data or content files.

However, the platform currently contains several fundamentally different
page categories:

1. Marketing pages
2. Product pages
3. Article pages (text content)
4. Documentation pages

Each of these page types has different structural and rendering needs.

Attempting to unify them under a single page engine would introduce unnecessary
abstraction and increase architectural complexity.

Instead, the project intentionally separates these systems and defines
clear architectural boundaries between them.

This ADR documents those boundaries and explains why the project avoids
introducing a universal page engine.

---

## Decision

The marketing platform uses four independent page systems.

Each system has its own rendering model and responsibilities.

## Marketing Pages

Marketing pages are standard Next.js pages built with custom layouts
and UI components.

Examples:

- `/`
- `/pricing`

These pages typically contain:

- hero sections
- feature highlights
- marketing copy
- calls to action
- pricing cards

Marketing pages are highly visual and layout-specific.

Because of this, they are authored manually and are not generated by a
content engine.

## Product Pages

Product pages describe individual products of the Software Forge ecosystem.

Examples:

- `/products/authforge`
- `/products/nextjs-test-kit`
- `/products/coming-soon`

Product pages reuse shared UI components such as:

- ProductHero
- ProductSection
- PricingCard
- PaymentButtons

These pages are semi-structured.

They reuse shared components but remain manually authored.

A generic product page system (such as `/products/[slug]`) may be introduced
in the future when the number of products grows significantly.

At the current stage the number of products is small, therefore manual pages
provide better clarity and simpler maintenance.

## Article Pages

Some pages contain mostly structured textual content.

Examples:

- `/legal`
- `/demo`
- `/privacy`
- `/refund`
- `/license`

These pages use the Article UI system located in:

`src/shared/ui/article`

Core components include:

- ArticleLayout
- ArticleTitle
- ArticleSection
- ArticleText

Example structure:

```tsx
<ArticleLayout>
  <ArticleTitle>Legal</ArticleTitle>
  <ArticleSection title="Terms of Service">
    <ArticleText>...</ArticleText>
  </ArticleSection>
</ArticleLayout>
```

This system standardizes typography and layout for text-heavy pages while
keeping the implementation simple and code-driven.

Content is intentionally not moved to data files or CMS systems.

The number of article pages is small, therefore introducing a content engine
would be unnecessary over-engineering.

## Documentation Pages

Documentation pages use a dedicated Markdown-based documentation engine.

Documentation content is stored in the repository under:

`content/authforge`

The documentation rendering pipeline is:

```text
Markdown
↓
Documentation renderer
↓
DocContent component
↓
Documentation layout
```

Documentation pages are served under routes such as:

`/en/docs/authforge/security`

The documentation system includes features such as:

- sidebar navigation
- page outline
- anchor navigation
- SEO metadata generation
- code block enhancements (such as optional copy buttons)

This documentation engine is intentionally isolated from the rest of the
marketing site architecture.

## Localization Strategy

The documentation system supports multi-language routing.

Documentation routes are structured as:

`/[locale]/docs/*`

Example URLs:

- `/en/docs/authforge/security`
- `/ru/docs/authforge/security`

Documentation translations are stored within the repository and bundled
with the product source.

The localization capability is primarily intended for developer
documentation, which may benefit from translations.

The marketing website itself currently does not implement full
localization.

Reasons include:

- the marketing copy is relatively small
- developer products are typically marketed in English
- localization infrastructure would increase project complexity

The current strategy is therefore:

- Marketing pages: English only
- Product pages: English only
- Documentation: multi-language supported

This approach mirrors many developer platforms where documentation may
be translated while the marketing site remains English-first.

## Rationale

Separating page systems provides several benefits:

- architectural clarity
- reduced abstraction
- easier maintenance
- predictable page structure
- independent evolution of documentation and marketing systems

Avoiding a universal page engine prevents premature generalization and
keeps the architecture aligned with the current scale of the project.

The documentation system already provides a structured content engine,
while other page types remain intentionally simple.

## Consequences

The project maintains separate rendering systems for:

- marketing pages
- product pages
- article pages
- documentation pages

Future architectural evolution may include:

- a product catalog system
- dynamic product pages
- extended documentation localization

However, such changes will only be introduced when the scale of the
platform justifies additional abstraction.

Until then the current architecture is considered stable and intentional.

## Notes

This ADR defines the architectural boundaries for page rendering
within the Software Forge marketing platform.

It does not affect backend APIs, billing systems, or product
implementation details.
